Ordering Rationale

  • Simplified data model and minimal lead API come first because the paid pilot depends on a working lead pipeline, not platform completeness.
  • Landing, CRM Kanban and manager notification are built in Sprint 1 to prove lead-to-CRM value immediately.
  • Operational CRM comes before Telegram and AI, because automation needs a usable manager workflow to update.
  • Telegram and n8n come before voice: they validate qualification and follow-up logic at lower operational cost.
  • Queues and workers are introduced before service extraction, because async load is the first scaling pressure.

MVP-first Rules

  • Build the smallest useful backend and PostgreSQL schema for Sprint 1.
  • Use n8n for early workflow velocity instead of custom orchestration.
  • Move Redis queues, API gateway, workers, advanced monitoring and rate limits to Sprint 5+.
  • Treat Docker, Sentry and hosting hardening as Sprint 1 stretch only.

Scale Gates

  • Split services only by load, ownership or release cadence.
  • Add event bus only when multiple consumers need durable domain events.
  • Add memory/vector stores only after agent quality depends on retrieval/context.
  • Add read replicas and tracing when analytics and service boundaries justify the overhead.
