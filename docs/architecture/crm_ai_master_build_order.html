Ordering Rationale

  • Data model, backend boundaries and hosting come first because every product surface depends on them.
  • CRM and landing are built before AI depth, because agents need real lead/pipeline state to act on.
  • Telegram and n8n come before voice: they validate qualification and follow-up logic at lower operational cost.
  • Queues and workers are introduced before service extraction, because async load is the first scaling pressure.

MVP-first Rules

  • Build one modular backend and one shared PostgreSQL database for MVP.
  • Use n8n for early workflow velocity instead of custom orchestration.
  • Stub Redis cache, admin/billing depth and future channels until pilots require them.
  • Keep monitoring minimal but present from the first paid pilot.

Scale Gates

  • Split services only by load, ownership or release cadence.
  • Add event bus only when multiple consumers need durable domain events.
  • Add memory/vector stores only after agent quality depends on retrieval/context.
  • Add read replicas and tracing when analytics and service boundaries justify the overhead.
